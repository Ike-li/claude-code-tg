# Code of Conduct

This project should be a practical, respectful space for people improving a
self-hosted developer tool.

## Scope

This code applies to project issues, pull requests, discussions, documentation,
release coordination, and maintainer interactions around `tgcc`. It applies to
contributors, users, maintainers, and anyone representing the project in a
community space.

## Expected Behavior

- Be respectful and assume good intent while still being precise.
- Keep feedback focused on the work, tradeoffs, evidence, and user impact.
- Welcome newcomers who are trying to reproduce an issue or make a small fix.
- Be careful with logs, screenshots, tokens, chat transcripts, and local paths.

## Unacceptable Behavior

- Harassment, threats, discriminatory language, or personal attacks.
- Publishing private credentials, Telegram chat contents, or exploit details.
- Repeated off-topic arguments that block maintainers from resolving issues.
- Pressuring maintainers or contributors to run unsafe configurations.

## Reporting

For security-sensitive reports, follow [SECURITY.md](SECURITY.md) and use
GitHub private vulnerability reporting.

Conduct reports are not accepted as public issue details. If the report includes
security-sensitive material, tokens, exploit details, chat transcripts, logs, or
local paths, use the private reporting path in [SECURITY.md](SECURITY.md). If
you already have another private maintainer contact for this project, use that
path and include:

- what happened and where
- who was involved
- whether the situation is ongoing
- any safety or privacy concerns that affect how maintainers should respond

If you cannot access a private contact, open only a minimal Question / help
issue asking maintainers to restore a private conduct contact path. That public
issue is routing only: do not include names, screenshots, private chat content,
tokens, logs, local paths, incident details, reporter identity, or impact
summary.

## Review Process

Maintainers should acknowledge conduct reports, keep details private, separate
security-sensitive material into the security reporting path, and review the
available context before taking action. People named in a report should not be
the only reviewer of that report when another maintainer is available.

## Enforcement

Maintainers may edit, hide, or close issues and pull requests that violate this
code. Depending on severity and history, maintainers may warn participants,
limit participation in a thread, temporarily block participation, or permanently block participation from the project.
